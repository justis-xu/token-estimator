# 一、背景：上下文引擎为什么需要 token 估算？

## 1.1 直接调 API 分词的延迟与成本代价

大语言模型按 token 计费，也按 token 限制上下文长度。要管好上下文窗口，就要知道每段文本占多少 token——这听起来是个小问题，但一旦放进实际系统，麻烦就来了。

最直接的做法是调分词 API：把文本发给模型提供商，让它返回准确的 token 数。这个数字绝对精确，但代价不小。

首先是**延迟**。分词 API 是一次网络请求，即便是内网调用也要几十毫秒。如果上下文引擎在每轮对话前都要计算一遍 prompt 的 token 数，这个等待会直接叠加到用户感知的响应时间上。

其次是**成本**。国内主流模型的分词接口多数独立计费或有调用频次限制。一个日活几千用户的系统，每轮对话触发一次分词请求，一天下来调用量相当可观。

更根本的问题是**可用性依赖**。分词 API 挂了，上下文引擎就没法工作——哪怕只是想做个简单的长度判断。

## 1.2 估算的精度要求：够用即可

既然直接调 API 有代价，就需要本地估算。但估算需要多准？

上下文引擎用 token 数做的决策主要是两类：一是判断当前上下文是否快要溢出，提前触发截断或压缩；二是在多条消息里排优先级，决定哪些先丢弃。

这两类决策都有一定容错空间。误差在 10%~15% 以内，引擎的判断不会出错——最多是截断触发得稍早或稍晚几轮，不影响正确性。

反过来说，估算不需要做到个位数精确。追求绝对精确意味着要在本地跑完整的 BPE 分词器，那和调 API 在复杂度上没有本质差别。

**目标因此很清晰：本地、无网络依赖、延迟接近零、误差控制在 15% 以内。**

# 二、四种方案横向对比

解决这个问题有几条路，每条都有人走过。

## 2.1 字符数 / 词数粗估

最简单的做法：中文按字符数 × 系数，英文按词数 × 系数。实现五行代码搞定，延迟接近零。

缺点也很明显——不同模型对同一段文本的分词结果差异很大，一个固定系数根本覆盖不了。对中英混合文本、代码、JSON 结构，误差可以轻松超过 30%。

## 2.2 CJK-aware 固定系数

改良版：把中文字符和英文字符分开处理，各自乘不同系数。比如「每个汉字算 1.5 个 token，每 4 个英文字母算 1 个 token」。

比方案一准一些，但本质问题没变——系数是拍的，没有针对具体模型校准，模型间差异仍然无法覆盖。而且中文标点、全角符号、日文假名这些字符怎么处理，大多数实现都没认真想过。

## 2.3 tiktoken 本地分词

OpenAI 开源了 tiktoken，可以在本地精确计算 GPT 系列模型的 token 数，不需要网络请求，速度很快。

局限是只支持 OpenAI 的模型编码（`cl100k_base`、`o200k_base`）。Qwen、DeepSeek、Claude 等模型有各自的词表，tiktoken 算出来的数字对它们来说是错的。国内模型为主的场景用不上。

## 2.4 直接调 API

如第一节所说，精确但有延迟、成本和可用性代价。适合离线批处理场景，不适合在线实时估算。

## 2.5 本方案：查表 + 全局 discount

核心思路是把分词计算拆成两个阶段：**离线生成词表，在线查表估算**。

离线阶段，用各模型的真实分词器处理 CJK 字符表中所有 20,992 个汉字，记录每个字对应的 token 数，存成一个紧凑的二进制文件（每个模型 20KB）。在线阶段，对输入文本逐字符分类，CJK 字符查表累加，其他字符类型用预设系数处理，最后乘以一个从真实语料校准出来的全局修正系数（discount，用于修正 BPE 跨字合并带来的系统性高估）。

> BPE（字节对编码）是大多数现代语言模型使用的分词算法，核心思路是把高频字节对反复合并成更大的词块。同一段文本在不同模型下产生的 token 数量差异，本质上来自各模型 BPE 词表的不同。

整个在线估算是纯内存操作，无网络、无 IO，延迟在微秒级。词表和系数文件可以独立更新，不需要重新编译部署。

## 3.1 两阶段设计总览

整个方案分两个阶段，关键在于把「重计算」全部推到离线，让在线路径只剩「查表」。

## 3.2 离线阶段

离线阶段的核心工作是为每个模型生成词表和 discount 系数，一次生成，长期复用。

### 3.2.1 词表三条生成路径及适用场景

不同模型的词表来源不同，因此生成脚本设计了三条路径：

| 路径 | 适用模型 | 原理 | 耗时 |
|------|---------|------|------|
| HuggingFace tokenizer | Qwen、DeepSeek、GLM、MiniMax、Kimi | 加载模型的分词器配置，本地逐字分词，只需几 MB 配置文件，无需下载模型权重 | 几秒到几分钟 |
| tiktoken | GPT-4o（`o200k_base`）、GPT-4（`cl100k_base`） | OpenAI 开源的本地分词库，完全离线 | 几秒 |
| 分词 API | 豆包、Claude | 逐字调接口取真实 token 数；受频率限制（豆包 10 次/秒，Claude 5 次/秒） | 30~70 分钟/张词表 |

### 3.2.2 语料构成

discount 系数从真实语料中计算，语料需要覆盖实际使用场景。目前使用三类来源混合：

- **中文维基百科**（50 篇随机摘要）：正式书面中文，高频汉字覆盖好
- **技术社区文章**（30 篇）：中英混合技术文本，贴近实际 prompt 风格
- **手写消息样本**（20 条）：覆盖 system 指令、user 问题、tool 输出三种消息格式，这三类在上下文引擎里最常见，手写比随机语料更贴合真实分布

### 3.2.3 discount 校准口径

discount 的计算逻辑：对语料中每条文本，分别用在线估算逻辑（discount 暂设为 1.0）和真实分词器计算 token 数，取比值 `真实值 / 估算值`，在所有文本上取平均。

这里有一个关键设计决策：discount 是对**整段估算值**的全局修正，而不是只修正 CJK 部分。原因是语料本身就是中英混合的，英文的启发式估算也存在系统性偏差，全局修正可以把两部分的偏差一起吸收进来。如果只修正 CJK 部分，混合文本的 discount 会因为分子（真实值）包含英文 token 而虚高，校准结果失真。

## 3.3 在线阶段：为什么查表能做到极低延迟

词表在服务启动时一次性读入内存，存成 `map[string][]byte`。之后每次估算只有三步：关键词匹配找到对应词表，单遍扫描文本做字符分类和累加，最后乘以系数取整。没有网络请求，没有文件 IO，没有复杂计算——整个过程在内存里完成，单次调用延迟在微秒量级。

9 个模型的词表合计约 180KB，完整加载进内存代价极低。

## 3.4 两阶段解耦：只换文件，无需重新编译

离线和在线之间只通过文件交换数据：`.bin` 词表文件和 `config.json` 系数文件。在线服务启动时读取这些文件，运行期间不再访问磁盘。

好处很直接：**词表和系数可以独立更新**。新增一个模型，或者根据线上数据重新校准 discount，只需要替换对应文件并重启服务，不需要改代码、不需要重新编译、不需要走发布流程。模型更新迭代快，这个解耦让维护成本大幅降低。

# 四、核心算法：字符分类分发

## 4.1 为什么要按字符类型分类

不同书写系统在 BPE 分词里的粒度差异很大。一个汉字通常是 1~3 个 token，一个英文单词按长度可以是 1~6 个 token，一个 emoji 可能是 3 个 token。用统一系数处理所有字符是方案一和方案二的问题所在——粒度混在一起，系数怎么选都不对。

在线估算的核心是一个单遍字符扫描，按字符类型分流到不同的处理逻辑。下面这张表是 8 种分类的完整一览：

```画图提示
详细信息表格图，共 8 行数据加一行表头，5 列。

表头（深色背景，白色文字）：
  字符类型 | Unicode 范围 | 处理逻辑 | 估算值 | 设计意图

数据行（交替浅灰/白色背景）：
1. CJK 基本区 | U+4E00~U+9FFF（20,992 字） | 查模型词表 | table[cp - 0x4E00]，精确到字 | 各模型对同一汉字的 token 数差异大，必须逐字查表
2. CJK 扩展区 | U+3400~U+4DBF，U+F900~U+FAFF | 兜底系数 | +1.5/字 | 生僻字，词表不覆盖，用保守估值
3. 拉丁字母连续段 | a-z，A-Z | 按词长分桶 | ceil(词长 / 4) | 英文 BPE 约每 4 字母合并为 1 token，长词自动增加
4. 中文标点 / 全角符号 | U+2000~U+206F，U+3000~U+303F，U+FF00~U+FFEF | 固定系数 | +1.0/字 | 标点通常独立成 token；曾错误归入 emoji 兜底导致高估，独立后修复
5. 日文假名 | U+3040~U+309F（平假名），U+30A0~U+30FF（片假名） | 固定系数 | +1.0/字 | 假名 BPE 粒度接近汉字
6. 韩文音节 | U+AC00~U+D7AF | 固定系数 | +1.5/字 | 韩文音节 BPE 切分较细
7. 数字连续段 | Unicode Nd 类 | 按位数 | ×0.5/位 | 数字通常多位合并为 1 token
8. ASCII 标点 | U+0021~U+007E 非字母数字 | 固定系数 | +0.7/个 | 常见标点多数独立成 token 或两两合并
   换行符 | \n \r | 固定系数 | +1.0/个 | 各模型对换行符处理一致
   其他（emoji 等） | 不在以上范围 | 兜底 | +3.0/个 | emoji 编码复杂，保守高估

整体风格：信息密度高，表格清晰，每列宽度合理，「设计意图」列字号略小以容纳更多文字，第 4 行（中文标点）用浅黄色背景突出，表示这是一个重要的踩坑修复。
```

## 4.2 BPE discount：为什么用全局乘子而不是只修正 CJK

查表得到的是每个 CJK 字符**单独出现**时的 token 数，但 BPE 分词是有上下文的——「中文」可能被合并成 1 个 token，而分开查「中」和「文」得到的是 2 个 token。这个「单字查表 vs 实际合并」的系统性偏差，就是 discount 要修正的。

discount 的计算方式是：在真实语料上，比较启发式估算值（discount=1.0）和真实 token 数的比值，取平均。discount < 1 说明查表在高估（正常，因为忽略了跨字合并）；discount 越小，说明该模型的合并率越高。

**为什么是全局乘子，而不是只修正 CJK 部分？** 因为校准用的语料本身是中英混合的，英文启发式（`ceil(词长/4)`）也存在系统性偏差。如果只对 CJK 部分乘 discount，混合文本里英文 token 的误差就没有出口——校准出来的系数也会因为分子包含英文真实 token 而虚高，整体更不准。全局乘子把所有字符类型的估算偏差一起收进来，校准口径和使用口径保持一致。

## 4.3 无词表时的降级策略

词表是可选的。如果某个模型的词表文件不存在，服务不会报错，而是按两级降级处理：

1. 优先使用豆包词表替代（豆包的分词粒度偏细，单字 token 数倾向于高估，做兜底不会漏掉 token，是安全的选择）
2. 如果豆包词表也没有，CJK 字符统一按 1.5 token/字 计算

discount 同理——如果 `config.json` 里没有对应模型的系数，回退到 `default` 键（取所有模型中最大的 discount 值，即最保守的修正），保证不会因为系数缺失导致估算结果为零。

这个设计让系统在「词表还没生成完」或「遇到未知新模型」时仍然可用，给出一个保守偏高的估算，而不是崩溃或返回错误。

# 五、精度、局限与适用边界

## 5.1 中文为主场景的参考误差范围

设计目标是在中文为主的混合文本上，平均绝对误差（MAE）控制在 15% 以内。从校准语料的测试来看，正式中文文本（维基百科类）通常能达到 5%~8%，中英混合技术文本约在 10%~15%。

这个精度对上下文引擎的典型决策（截断触发、优先级排序）已经足够。

## 5.2 英文 / 代码场景的局限

英文没有字符级词表，只有启发式 `ceil(词长/4)` 加全局 discount 修正。对规律的英文散文，误差还可以接受；对代码、变量名、专有名词，误差会明显上升——`getUserByIdAndStatus` 这种词，实际可能被分成 7~8 个 token，而 `ceil(22/4)=6` 只算了 6 个，discount 修正后可能更低。

纯英文或代码密集的场景，这套方案的精度不如 tiktoken 本地分词（后者对 GPT 系列可以做到精确）。

## 5.3 何时放弃估算、直接调 API

以下场景建议直接调分词 API，不要用估算：

- **精确计费**：按 token 数核算费用，误差不可接受
- **接近窗口上限的判断**：上下文已经用到 95% 以上，几百 token 的误差就可能导致超限
- **纯英文 / 纯代码内容**：启发式精度不足，tiktoken（GPT）或对应模型 API 更合适

## 5.4 上线后如何持续校准系数

初始 discount 系数从有限语料计算，不一定能完全反映线上流量的真实分布。上线后可以按以下方式持续优化：

记录每次 API 调用返回的 `usage.prompt_tokens`（真实值）和当时的估算值，积累 500 条以上后，重新跑校准脚本，生成新的 `config.json`，替换文件并重启服务即可生效——不需要改代码、不需要重新编译。

积累越多真实样本，discount 越贴近实际流量分布，估算精度会持续提升。
