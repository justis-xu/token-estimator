# 一、背景：

> 项目开源地址：[github.com/justis-xu/token-estimator](https://github.com/justis-xu/token-estimator)

## 1.1 直接调 API 分词的延迟与成本代价

大语言模型按 token 计费，也按 token 限制上下文长度。要管好上下文窗口，就要知道每段文本占多少 token——这听起来是个小问题，但一旦放进实际系统，麻烦就来了。

最直接的做法是调分词 API：把文本发给模型提供商，让它返回准确的 token 数。这个数字绝对精确，但代价不小。

首先是**延迟**。分词 API 是一次网络请求，即便是内网调用也要几十毫秒。如果上下文引擎在每轮对话前都要计算一遍 prompt 的 token 数，这个等待会直接叠加到用户感知的响应时间上。

其次是**成本**。国内主流模型的分词接口多数独立计费或有调用频次限制。一个日活几千用户的系统，每轮对话触发一次分词请求，一天下来调用量相当可观。

更根本的问题是**可用性依赖**。分词 API 挂了，上下文引擎就没法工作——哪怕只是想做个简单的长度判断。

## 1.2 估算的精度要求：够用即可

既然直接调 API 有代价，就需要本地估算。但估算需要多准？

上下文引擎用 token 数做的决策主要是两类：一是判断当前上下文是否快要溢出，提前触发截断或压缩；二是在多条消息里排优先级，决定哪些先丢弃。

这两类决策都有一定容错空间。误差在 10%~15% 以内，引擎的判断不会出错——最多是截断触发得稍早或稍晚几轮，不影响正确性。

反过来说，估算不需要做到个位数精确。追求绝对精确意味着要在本地跑完整的 BPE 分词器，那和调 API 在复杂度上没有本质差别。

**目标因此很清晰：本地、无网络依赖、延迟接近零、误差控制在 15% 以内。**

## 2.1 字符数 / 词数粗估

最简单的做法：中文按字符数 × 系数，英文按词数 × 系数。实现五行代码搞定，延迟接近零。

缺点也很明显——不同模型对同一段文本的分词结果差异很大，一个固定系数根本覆盖不了。对中英混合文本、代码、JSON 结构，误差可以轻松超过 30%。

## 2.2 CJK-aware 固定系数

改良版：把中文字符和英文字符分开处理，各自乘不同系数。比如「每个汉字算 1.5 个 token，每 4 个英文字母算 1 个 token」。

比方案一准一些，但本质问题没变——系数是拍的，没有针对具体模型校准，模型间差异仍然无法覆盖。而且中文标点、全角符号、日文假名这些字符怎么处理，大多数实现都没认真想过。实测中文模型均值约 1.0 token/字，而非通常假设的 1.5——固定系数越精细也越容易出错。

## 2.3 tiktoken 本地分词

OpenAI 开源了 tiktoken，可以在本地精确计算 GPT 系列模型的 token 数，不需要网络请求，速度很快。

局限是只支持 OpenAI 的模型编码（`cl100k_base`、`o200k_base`）。Qwen、DeepSeek、Claude 等模型有各自的词表，tiktoken 算出来的数字对它们来说是错的。国内模型为主的场景用不上。

## 2.4 直接调 API

如第一节所说，精确但有延迟、成本和可用性代价。适合离线批处理场景，不适合在线实时估算。

## 2.5 本方案：单字表 + 高频词表 + 分段 discount

核心思路是把分词计算拆成两个阶段：**离线生成词表，在线查表估算**。

离线阶段分三步走：

1. **单字表**：对 CJK 字符块（U+4E00–U+9FFF，20,992 个汉字）逐字调用各模型分词器，记录每字对应的 token 数，存成 20KB 的二进制文件。

2. **高频词表**：从约 1400 万字的真实中文语料中提取出现频率最高的 5000 个相邻汉字对（即最常用的双字词），同样查询分词器，记录每个词的实际 token 数。这是对 BPE 合并现象的精确修正——「中国」在多数模型里是 1 个 token，但单字查表会估成 2 个，而高频词表能直接命中这类高频合并。

3. **分段 discount**：按文本中的中文字符比例将文本分为三类（纯中文 ≥ 60%、中英混合、纯英文 ≤ 10%），分别从语料中校准独立的修正系数。同等文本里，纯中文场景 BPE 合并率更高，需要更大的折扣；中英混合和纯英文场景的 discount 明显不同，不该用同一个系数混着算。

> BPE（字节对编码）是大多数现代语言模型使用的分词算法，核心思路是把高频字节对反复合并成更大的词块。同一段文本在不同模型下产生的 token 数量差异，本质上来自各模型 BPE 词表的不同。

在线阶段，单字表和高频词表在服务启动时一次性加载（12 个模型共约 370KB）。每次估算，CJK 相邻两字优先查高频词表，命中就跳 2 字，查不到再退化到单字表，最后乘以对应文本类别的 discount 取整。整个过程纯内存，无网络，无 IO，延迟在微秒级。

# 三、设计全貌

## 3.1 两阶段设计总览

整个方案分两个阶段，关键在于把「重计算」全部推到离线，让在线路径只剩「查表」。

<img src="https://img-1302474103.cos.ap-nanjing.myqcloud.com/202606280823261.png" style="zoom:50%;" />

## 3.2 离线阶段

离线阶段的核心工作是为每个模型生成词表和 discount 系数，一次生成，长期复用。

### 3.2.1 词表三条生成路径及适用场景

不同模型的词表来源不同，因此生成脚本设计了三条路径：

| 路径                  | 适用模型                                                                    | 原理                                                                     | 耗时              |
| --------------------- | --------------------------------------------------------------------------- | ------------------------------------------------------------------------ | ----------------- |
| HuggingFace tokenizer | Qwen、DeepSeek、GLM、MiniMax、Kimi                                          | 加载模型的分词器配置，本地逐字分词，只需几 MB 配置文件，无需下载模型权重 | 几秒到几分钟      |
| tiktoken              | GPT-4o（`o200k_base`）、GPT-4（`cl100k_base`）、Claude（`o200k_base` 近似） | OpenAI 开源的本地分词库，完全离线                                        | 几秒              |
| 分词 API              | 豆包                                                                        | 逐字调接口取真实 token 数；受频率限制（豆包 40 次/秒）                   | 30~70 分钟/张词表 |

### 3.2.2 语料构成

校准工作需要两份语料：**高频词统计语料**（词频统计越准，覆盖的高频合并越多）和 **discount 校准语料**（用于计算估算值与真实值的偏差比）。两者复用同一份语料文件。

语料来源四类，合计约 1400 万字：

- **中文维基百科**（~4000 篇，全文，每篇最多 8000 字，via HF `wikimedia/wikipedia`）：正式书面中文，高频汉字和词语组合覆盖全面
- **知乎 KOL**（~1000 条，via HF `wangrui6/Zhihu-KOL`）：真实中文问答，覆盖口语化表达和专业领域
- **英文维基百科**（~200 篇，via HF `wikimedia/wikipedia`）：补充英文和中英混排场景
- **OpenWebText**（~300 条，via HF `Skylion007/openwebtext`）：英文网页文本，贴近真实 prompt 分布
- **手写消息样本**（52 条）：覆盖 system 指令、user 问题、tool 输出、代码块、JSON 结构等格式，手写比随机语料更贴合真实分布

### 3.2.3 高频词表的生成

从语料中统计所有相邻汉字对的出现频率，取频率最高的 5000 个，对每个词查询各模型分词器，记录实际 token 数，存成紧凑二进制文件（每模型约 25KB）。

在线估算时，扫描到两个相邻 CJK 字符时优先查高频词表：命中则直接使用查到的 token 数并跳过这两个字，查不到再退化到单字表逐字查。高频词（「的」「了」「人」「中国」「发展」这类）覆盖率很高，能消除大多数单字高估。

### 3.2.4 分段 discount 校准

全局 discount 的问题在于：纯中文文本和中英混合文本的 BPE 合并率差异很大，一个系数两边都照顾不好。

现在按文本中的 CJK 字符占比把语料分成三桶：

| 类别    | CJK 比例 | 代表场景                     |
| ------- | -------- | ---------------------------- |
| `zh`    | ≥ 60%    | 系统提示、纯中文问答、新闻   |
| `mixed` | 10%–60%  | 技术文章、代码解释、中英混排 |
| `en`    | ≤ 10%    | 纯英文 prompt、代码块        |

对每类文本单独收集比值 `真实 token 数 / 启发式估算值`，取第 55 百分位作为该类的 discount。选 55 百分位而非均值的原因是：在 context-compression 场景里，**低估比高估代价更大**（低估导致 context 溢出，高估只是多截一点），略微保守的系数是合理的安全边际。

## 3.3 在线阶段：为什么查表能做到极低延迟

单字表和高频词表在服务启动时一次性读入内存。之后每次估算：关键词匹配找到对应词表，单遍扫描文本——CJK 相邻两字先查高频词表，命中跳 2 字，否则查单字表，最后乘分段 discount 取整。没有网络请求，没有文件 IO，没有复杂计算——整个过程在内存里完成，单次调用延迟在微秒量级。

12 个模型的单字表 + 高频词表合计约 370KB，完整加载进内存代价极低。

## 3.4 两阶段解耦：只换文件，无需重新编译

离线和在线之间只通过文件交换数据：`.bin` 词表文件和 `config.json` 系数文件。在线服务启动时读取这些文件，运行期间不再访问磁盘。

好处很直接：**词表和系数可以独立更新**。新增一个模型，或者根据线上数据重新校准 discount，只需要替换对应文件并重启服务，不需要改代码、不需要重新编译、不需要走发布流程。模型更新迭代快，这个解耦让维护成本大幅降低。

# 四、核心算法：字符分类分发

## 4.1 为什么要按字符类型分类

不同书写系统在 BPE 分词里的粒度差异很大。一个汉字通常是 1~3 个 token，一个英文单词按长度可以是 1~6 个 token，一个 emoji 可能是 3 个 token。用统一系数处理所有字符是方案一和方案二的问题所在——粒度混在一起，系数怎么选都不对。

在线估算的核心是一个单遍字符扫描，按字符类型分流到不同的处理逻辑。以下按处理策略分四类，覆盖所有字符场景：

<img src="https://img-1302474103.cos.ap-nanjing.myqcloud.com/202606281103333.png" style="zoom:50%;" />

## 4.2 两层修正：高频词表先近似捕捉，discount 再修正剩余误差

单字查表的系统性问题是：BPE 分词有上下文，「中国」在多数模型里是 1 个 token，但单字查「中」加查「国」得到 2 个，高估了。

**第一层修正：高频词表**。对 5000 个高频词直接查表得到 token 数，在线估算时优先命中，命中则跳 2 字不再查单字表。高频词的 token 数在生成阶段孤立 tokenize（`encode("中国")`），不再受 discount 缩放——它的误差源是独立的（见下文），用统计系数去抵消会引入新的偏差。

> **高频词表的精度边界**：这里的"精确"是近似意义上的精确，不是零误差。有两个内生的局限：第一，词表是孤立 tokenize 出来的，但 BPE 合并依赖上下文——「中国」独立 tokenize 是 1 个 token，但在「中国人民」这个字符串里，合并边界可能不一样；第二，在线估算是贪心左到右匹配，「ABCD」里如果 AB、BC、CD 都在词表里，贪心取 AB+CD，但真实分词可能以 BC 为边界。这两个误差随机且互相独立，大量文本上趋于抵消，是可以接受的系统性近似。

**第二层修正：分段 discount**。高频词表未覆盖的低频词对、英文启发式的偏差，由分段 discount 统一兜底。discount 只乘在高频词表**未命中**的部分（单字表 + 英文启发式 + 其余字符）：

```
最终估算 = 高频词表命中 token 数 + 其余启发式估算 × discount
```

校准方式与这个公式严格对齐：对每类文本（zh / mixed / en），收集 `(真实 token 数 - 高频词命中数) / 启发式估算值` 的比值分布，取第 55 百分位作为 discount。这样避免了用统计系数去"纠正"已经相对精确的高频词命中，校准口径与生产公式保持一致。

**为什么选 55 百分位而非均值？** 在 context-compression 场景，低估比高估代价更大：低估可能导致真实 token 数超出 context 窗口，而高估只是多截掉一点历史。选略高于均值的 55 百分位，是对这个不对称风险的主动对冲。

## 4.3 无词表时的降级策略

词表是可选的。如果某个模型的词表文件不存在，服务不会报错，而是自动降级：CJK 扩展区的生僻字仍然硬编码为 3.0 token/字（UTF-8 三字节，不依赖词表），基本区汉字按除数 1.5 换算为约 0.67 token/字，该值来自 12 个主流模型的实测均值，偏保守不会低估；其余字符完全沿用英文和标点的启发式规则；discount 则回退到所有模型中最保守的 default 系数。

这个设计让系统在「词表还没生成完」或「遇到未知新模型」时仍然可用，给出一个偏高的保守估算，而不是崩溃或返回错误。若需要更准确的结果，只需运行 `generate_tables.py` 为新模型生成专属词表（HuggingFace 模型几分钟内完成）。

# 五、精度、局限与适用边界

## 5.1 中文为主场景的参考误差范围

设计目标是在中文为主的混合文本上，平均绝对误差（MAE）≤ 15%。三层设计的预期效果：

- **单字表**：初步估算，纯中文文本 MAE 约 15%~20%（因 BPE 合并被高估）
- **+ 高频词表**：覆盖高频词后，纯中文 MAE 降至约 8%~12%
- **+ 分段 discount**：按文本类别独立修正后，各类别 MAE 均控制在 10% 以内

基于 68,698 条 golden 样本（12 个模型，每模型约 5,725 条）的实测结果：

| 模型                   | MAE  |
| ---------------------- | ---- |
| gpt-4                  | 2.7% |
| gpt-4o / claude        | 5.2% |
| glm / glm4             | 7.0% |
| qwen2                  | 6.6% |
| deepseek / deepseek-v3 | 7.3% |
| doubao                 | 7.5% |
| qwen                   | 7.7% |
| minimax                | 8.6% |
| kimi                   | 9.1% |

全部模型 MAE ≤ 15%，大多数在 10% 以内。这个精度对上下文引擎的典型决策（截断触发、优先级排序）已经足够。

基于真实网关抽样复核后，本地估算在以下 6 个模型上的 MAE 如下：

| 模型                | 样本数 | MAE    |
| ------------------- | ------ | ------ |
| `deepseek-v4-flash` | 24     | 8.35%  |
| `qwen3.7-plus`      | 24     | 11.14% |
| `minimax-m3`        | 24     | 7.23%  |
| `minimax-m2.7`      | 24     | 7.39%  |
| `glm-5.1`           | 24     | 9.98%  |
| `kimi-k2.5`         | 24     | 10.60% |

这个精度足够做上下文预算预警，但不适合拿去做精确计费。

## 5.2 英文 / 代码场景的局限

英文没有字符级词表，只有启发式 `ceil(词长/4)` 加全局 discount 修正。对规律的英文散文，误差还可以接受；对代码、变量名、专有名词，误差会明显上升——`getUserByIdAndStatus` 这种词，实际可能被分成 7~8 个 token，而 `ceil(22/4)=6` 只算了 6 个，discount 修正后可能更低。

纯英文或代码密集的场景，这套方案的精度不如 tiktoken 本地分词（后者对 GPT 系列可以做到精确）。

## 5.3 何时放弃估算、直接调 API

以下场景建议直接调分词 API，不要用估算：

- **精确计费**：按 token 数核算费用，误差不可接受
- **接近窗口上限的判断**：上下文已经用到 95% 以上，几百 token 的误差就可能导致超限
- **纯英文 / 纯代码内容**：启发式精度不足，tiktoken（GPT）或对应模型 API 更合适

## 5.4 上线后如何持续校准系数

初始 discount 系数从有限语料计算，不一定能完全反映线上流量的真实分布。上线后可以按以下方式持续优化：

记录每次 API 调用返回的 `usage.prompt_tokens`（真实值）和当时的估算值，积累 500 条以上后，重新跑校准脚本，生成新的 `config.json`，替换文件并重启服务即可生效——不需要改代码、不需要重新编译。

积累越多真实样本，discount 越贴近实际流量分布，估算精度会持续提升。
